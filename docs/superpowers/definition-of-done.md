# 完成定义标准 / Definition of Done

## 目的

本文档定义 MiniMall 项目中所有任务完成的统一标准。所有 Agent 在将任务标记为完成前，必须满足本清单中的所有条件。

---

## 验收标准

### 1. 代码质量 / Code Quality

- [ ] 代码能通过 `mvn checkstyle:check`
- [ ] 代码能通过 `mvn spotless:check`（如适用）
- [ ] 无 merge conflict markers（`<<<<<<<`, `=======`, `>>>>>>>`）
- [ ] 代码遵循项目的命名规范和代码风格
- [ ] 函数长度不超过 50 行，文件大小不超过 800 行
- [ ] 无深度嵌套（不超过 4 层），使用 early return 模式

### 2. 测试 / Testing

- [ ] 新功能包含单元测试
- [ ] 关键业务逻辑包含集成测试
- [ ] `mvn test` 全部通过
- [ ] 测试覆盖率不低于 80%
- [ ] 测试遵循 AAA（Arrange-Act-Assert）模式
- [ ] 测试命名具有描述性，说明被测试的行为

### 3. 安全性 / Security

- [ ] 无硬编码凭证（API keys, passwords, tokens）
- [ ] 用户输入已验证
- [ ] SQL 注入防护（使用参数化查询）
- [ ] 无 XSS 漏洞（用户输入已转义）
- [ ] 错误消息不泄露敏感信息

### 4. 错误处理 / Error Handling

- [ ] 错误被显式处理，不是静默失败
- [ ] 日志级别正确使用：
  - `log.error`: 仅用于真正的错误情况
  - `log.warn`: 用于可恢复的问题
  - `log.info`: 用于重要业务事件
- [ ] 面向用户的错误有友好的提示信息

### 5. 业务逻辑 / Business Logic

- [ ] 业务规则实现正确
- [ ] 边界条件已处理
- [ ] 事务边界清晰（如涉及数据库操作）
- [ ] 并发场景已考虑（如适用）

### 6. 文档更新 / Documentation

- [ ] README.md 已更新（如影响项目说明）
- [ ] API 文档已更新（如涉及 API 变更）
- [ ] 必要注释已添加（仅添加 WHY，不添加 WHAT）
- [ ] 本文档（如适用）已更新

### 7. Git 规范 / Git Standards

- [ ] 提交信息符合规范（type: description）
- [ ] 分支名称符合规范
- [ ] 无不必要的文件提交
- [ ] 提交前已执行代码合并检查脚本

### 8. 功能完整性 / Feature Completeness

- [ ] 功能符合原始需求
- [ ] 验收标准已满足
- [ ] 在目标环境测试通过
- [ ] 相关联的功能未被破坏

---

## 流程

1. **开发前**: 阅读并理解本清单
2. **开发中**: 按清单标准开发
3. **提交前**: 执行自查，确保所有条件满足
4. **Code Review**: 请求代码审查，审查者按本清单验证
5. **合并后**: 确认任务状态更新为 Done

---

## 术语定义

| 术语 | 定义 |
|------|------|
| 单元测试 | 针对单个函数或方法的测试 |
| 集成测试 | 针对多个组件交互的测试 |
| Early Return | 提前返回，减少嵌套的编程模式 |
| AAA 模式 | Arrange（准备）→ Act（执行）→ Assert（断言） |
