# 代码提交检查清单

## 目的

防止 agent 声称完成但代码未提交到代码库的问题再次发生。每个 issue 完成前必须通过所有检查项。

---

## 检查项清单

### 1. 代码提交验证

- [ ] **G1**: 代码已通过 `git status` 确认有变更
- [ ] **G2**: 代码已通过 `git diff` 确认变更内容正确
- [ ] **G3**: 代码已通过 `git log --oneline -5` 确认提交历史正常

### 2. 分支与远程验证

- [ ] **R1**: 当前分支正确 (`git branch` 显示在正确的分支上)
- [ ] **R2**: 远程仓库已配置 (`git remote -v` 显示正确的 origin)
- [ ] **R3**: 远程分支存在或已创建 (`git ls-remote --heads origin` 验证)

### 3. 提交信息规范

- [ ] **C1**: 提交信息符合规范格式: `<type>: <description>`
- [ ] **C2**: 提交信息描述清晰,说明本次做了什么
- [ ] **C3**: 提交信息不包含敏感信息或调试内容

### 4. 代码质量检查

- [ ] **Q1**: 代码已通过本地构建 (`mvn compile` 或等效命令)
- [ ] **Q2**: 没有新增的编译警告或错误
- [ ] **Q3**: 没有遗留的调试代码 (`console.log`, `System.out.println` 等)

### 5. 测试验证

- [ ] **T1**: 核心业务逻辑有对应的单元测试
- [ ] **T2**: 测试通过 (`mvn test` 或等效命令)
- [ ] **T3**: 测试覆盖率满足要求 (80%+)

### 6. 文档更新

- [ ] **D1**: 代码变更已更新相关文档
- [ ] **D2**: 新增功能已添加到 README 或用户指南
- [ ] **D3**: API 变更已更新 API 文档

### 7. CI/CD 检查

- [ ] **CI1**: CI 流水线状态为通过 (如适用)
- [ ] **CI2**: 没有阻塞合并的问题

### 8. Issue 状态

- [ ] **S1**: Issue 所有验收标准已满足
- [ ] **S2**: Issue 状态已更新为 `in_review` 或 `done`

---

## 执行流程

### Agent 完成工作后的检查流程

```
1. 运行 git status 确认有变更
2. 运行 git diff 确认变更内容
3. 确认分支正确
4. 确认远程仓库配置正确
5. 提交代码: git add . && git commit -m "feat: description"
6. 推送代码: git push -u origin HEAD
7. 运行本地构建验证
8. 运行测试验证
9. 更新相关文档
10. 在 Multica Issue 中更新状态为 in_review
11. 在 Issue 下发布完成评论,说明已完成的工作和验证结果
```

### 验收流程 (Reviewer)

```
1. 检查提交记录是否存在
2. 检查提交信息是否符合规范
3. 检查代码变更是否与 Issue 要求一致
4. 检查测试是否通过
5. 检查 CI 状态
6. 检查文档是否更新
7. 确认所有检查项通过后,将 Issue 状态更新为 done
```
