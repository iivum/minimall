# Issue 管理规范 / Issue Management Standards

## 目的

减少重复 issue 的创建，提高团队协作效率，确保每个 issue 都有明确的执行者和验收标准。

---

## 1. Issue 创建规范

### 1.1 创建前必查

**创建新 issue 前，必须执行以下检查：**

1. 使用 `multica issue list --status todo --limit 100 --output json` 搜索现有 issues
2. 检查是否有相同或相似的 open issue
3. 搜索关键词确认无重复

### 1.2 Issue 标题规范

```
[问题域]: [简洁描述]
```

示例：
- ✅ `CI/CD: 完善 merge gate 验证流程`
- ❌ `完善验证流程`

### 1.3 Issue 内容模板

```markdown
## Goal
[一句话描述目标]

## 背景
[为什么需要做这件事，关联的阻塞问题]

## 任务内容
1. [具体任务]
2. [具体任务]

## 验收标准
- [可验证的验收条件]
- [可验证的验收条件]

## 优先级: P0/P1/P2
## 执行者: [明确 assignee]
```

### 1.4 禁止事项

- ❌ 禁止创建无 assignee 的 issue
- ❌ 禁止创建无验收标准的 issue
- ❌ 禁止创建与现有 open issue 重复的 issue
- ❌ 禁止创建标题过于模糊的 issue（如"优化"、"完善"、"解决"）

## 2. Issue 去重机制

### 2.1 去重检查清单

创建 issue 前，对照以下清单检查：

| 检查项 | 操作 |
|--------|------|
| 相同问题 | 搜索现有 open issues，确认无相同问题 |
| 相似的子问题 | 如果是父 issue 的子任务，链接到父 issue |
| 历史问题 | 检查已完成的相似 issues，避免重复 |
| 阻塞关系 | 确认是否为某问题的阻塞项，不是则关联 |

### 2.2 发现重复 issue 处理流程

1. **识别重复**：发现现有 issue 与新 issue 内容重复
2. **合并策略**：
   - 如果现有 issue 状态为 `todo` 且 assignee 已确认执行，则直接在该 issue 下添加子任务
   - 如果现有 issue 长期未执行且无进展，标记为 `blocked` 并说明原因
   - 如果现有 issue 已完成，新 issue 确认为新需求则创建
3. **文档记录**：在评论中记录去重决策和合并原因

### 2.3 已识别重复 issues 处理建议

| 重复组 | 建议 |
|--------|------|
| AppID 问题 (MIN-1408, 1367, 1362, 1312, 1412, 1472, 1510) | 保留最新且描述最完整的，关闭其余 |
| 文档更新 (MIN-1477, 1535) | 合并为一个，分配给同一执行者 |
| CI/CD 流程 (MIN-1518, 1572) | 确认依赖关系，要么合并要么明确父子关系 |
| 技术债清理 (MIN-1319, 1534) | 合并为单一 backlog item |
| 团队驱动 (MIN-1398, 1365, 1316, 1584) | 保持一个 active，其余关闭 |

## 3. Issue 状态管理

### 3.1 状态流转

```
backlog → todo → in_progress → in_review → done
                  ↓
               blocked
```

### 3.2 状态变更规则

| 状态 | 变更条件 |
|------|----------|
| `todo` | 创建 issue 时的初始状态 |
| `in_progress` | 开始执行时必须设置 |
| `blocked` | 遇到阻塞时立即设置，并在评论中说明原因 |
| `in_review` | PR 合并后或待审查时设置 |
| `done` | 验收标准全部满足时设置 |

### 3.3 验收前置条件

Issue 标记为 `done` 前必须满足：
- ✅ 所有验收标准已满足
- ✅ 相关文档已更新
- ✅ 代码已合并到目标分支

---

## 4. Sprint Issue 规范

### 4.1 Sprint Issue 命名

```
Sprint #XX: [具体任务描述]
```

### 4.2 Sprint Issue 检查清单

- [ ] 明确属于哪个 Sprint
- [ ] 有具体的任务内容和验收标准
- [ ] 分配给明确的执行者（非 "团队"）
- [ ] 无重复的父 issue
