# Definition of Done (DoD)

**版本**: 1.0
**创建日期**: 2026-05-13
**更新日期**: 2026-05-13
**维护者**: Orion

---

## 一、概述

本文档定义了 Minimall 项目中所有工作项的完成标准。DoD 确保团队对"完成"有一致的理解，从而提高质量和协作效率。

---

## 二、代码合并 DoD

所有代码合并到 main 分支前，必须满足以下所有条件:

### 2.1 CI/CD 检查

| 检查项 | 要求 | 失败处理 |
|--------|------|----------|
| Merge conflict detection | 无冲突标记 | 阻止合并 |
| Build | 编译成功 | 阻止合并 |
| Security scan | 无高危漏洞 | 阻止合并 |
| Code quality | checkstyle 通过 | 警告 |
| Test coverage | ≥80% | 阻止合并 |

### 2.2 代码审查

- [ ] 代码经过 code-reviewer agent 审查
- [ ] 无 CRITICAL 或 HIGH 问题
- [ ] 安全敏感代码经过 security-reviewer agent 审查

### 2.3 代码质量

- [ ] 无未解决的 TODO/FIXME (>10 触发警告)
- [ ] 无 hardcoded secrets
- [ ] 无 console.log 或 debug 语句
- [ ] 函数 <50 行
- [ ] 文件 <800 行
- [ ] 无深度嵌套 (>4 层)

---

## 三、功能完成 DoD

所有功能 Issue 完成前，必须满足以下所有条件:

### 3.1 开发完成

- [ ] 前端和后端代码均已实现
- [ ] API 文档已更新
- [ ] 功能测试通过

### 3.2 联调验证

- [ ] 前后端联调测试通过
- [ ] 集成测试通过 (如适用)

### 3.3 文档更新

- [ ] README.md 已更新 (如适用)
- [ ] API 文档已更新
- [ ] 关键决策已记录

### 3.4 验收

- [ ] 产品验收通过
- [ ] 阻塞问题已升级或解决

---

## 四、文档 DoD

所有文档创建/更新完成前，必须满足以下所有条件:

- [ ] 内容准确、完整
- [ ] 格式符合项目规范
- [ ] 无拼写/语法错误
- [ ] 相关人员已 review

---

## 五、Sprint DoD

每个 Sprint 结束前，必须满足以下所有条件:

### 5.1 交付完成

- [ ] 所有 P0/P1 Issue 已完成
- [ ] Sprint 完成率 >85%
- [ ] 无未解决的阻塞项

### 5.2 流程改进

- [ ] 本次 Sprint 回顾会议已召开
- [ ] 改进建议已文档化
- [ ] 下次 Sprint 计划已确定

---

## 六、Issue 完成检查清单

执行者完成任务后，对照以下清单自检:

```
□ CI/CD 所有检查通过
□ code-reviewer agent 审查通过
□ 无安全漏洞
□ 测试覆盖率 ≥80%
□ 代码符合质量标准
□ API 文档已更新
□ 前后端联调通过
□ 产品验收通过
□ 相关文档已更新
□ 无阻塞项
```

---

## 七、持续改进

- **季度回顾**: 每季度回顾 DoD 执行情况
- **数据驱动**: 根据 Sprint 数据调整标准
- **团队共识**: 所有变更需团队确认
